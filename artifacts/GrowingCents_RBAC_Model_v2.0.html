RBAC Model v2.0 — 7-Layer Interaction Design

This document defines the complete Role-Based Access Control model as requested by Moses Aaku — covering all 7 layers, how they interact, inheritance and propagation, runtime flexibility without coding, multi-role assignment, and honest known gaps.

1. The 7-Layer Model

LayerConceptGrowing Cents ExampleDescription
7UserMoses Aaku, Grace Atim, Emmanuel MvanoAn actual person. Has no permissions of their own — inherits everything through role assignments. User with no role = zero access (orphaned).
6RoleClub Admin, Club Member, Auditor, Club SecretaryA business-facing job function. A named bundle of permission groups. Managed at runtime via admin panel — no coding needed.
5Permission GroupReporting Permissions, Assessment PermissionsA logical bundle of related permissions used as a building block for roles. In Spatie: multiple named permissions assigned to a role in one operation.
4Permissionview_reports, submit_assessment, export_dataThe exact allowed right: Action ON Resource. A named string checked at middleware and controller level.
3ActionView, Submit, Export, Manage, DownloadWhat the user wants to DO. Maps to HTTP methods and controller methods.
2ResourceAssessment, Report, Club Profile, Audit LogThe THING being accessed. Further scoped by club_id to enforce data isolation.
1Policy / ContextActive subscription, 7-day grace period, status = activeConditions layered ON TOP of permissions. A user may have view_reports but still be blocked if no active subscription.

2. Inheritance & Propagation

Removing a Permission From a Role

When Admin removes view_reports from the Club Member role via the admin panel:

  1. Spatie updates role_has_permissions — removes the link
  2. ALL users who hold Club Member role immediately lose view_reports access
  3. No per-user changes needed — permission is evaluated dynamically on each request
  4. Club Admins and Super Admins are unaffected — their role records are unchanged

Orphaned Access — User With No Role

  • model_has_roles: no entries for this user
  • Every permission check returns false → every protected route returns 403
  • User can reach the login page but cannot access any resource after login

3. Runtime Role & Permission Management — No Coding Required

Scenario: Create 'Club Secretary' Role

Admin wants a new role that can only generate/view reports — no submission rights:

  1. Navigate to Admin Panel → User Management → Roles → Create Role
  2. Enter name: Club Secretary. Check permissions: ✅ view_reports ✅ export_data
  3. Click Save → UserManagementController@storeRole() runs: Role::create(['name'=>'Club Secretary']) then $role->syncPermissions(['view_reports','export_data'])
  4. Navigate to Users → Find the user → Edit → Assign Role: Club Secretary
  5. User now has view_reports and export_data — cannot submit assessments
Zero code changes. Zero deployment. Entire scenario handled in the admin panel UI.

Scenario: New Platform Feature → Assign to Role

  1. Developer adds new permission to DB on feature deployment (one-time migration)
  2. Admin navigates to Roles → Edit relevant role → Check new permission → Save
  3. All users with that role immediately inherit the new permission

4. Multi-Role Assignment

Assign Multiple Roles Simultaneously

// Assign both Club Admin AND Auditor roles
$user->syncRoles(['Club Admin', 'Auditor']);

// Add role without removing existing
$user->assignRole('Auditor'); // keeps Club Admin, adds Auditor

When a user holds multiple roles, Spatie merges all permissions — the user gets the union of all permissions across all roles.

Promote Member to Admin in a Specific Club

// Step 1: Update Spatie system role
$user->syncRoles('Club Admin'); // removes Club Member, assigns Club Admin

// Step 2: Update club pivot role
$club->users()->updateExistingPivot($userId, ['role' => 'admin']);

5. Club-Specific vs Platform-Wide — The Known Tension

Gap #6: Spatie roles are platform-wide. If Grace is "Club Admin" in Spatie, she has Club Admin permissions across all clubs at the permission layer. Data isolation (club_id scoping in controllers) prevents her from seeing other clubs' data, but the permission itself isn't club-specific.

Current Approach

  • Spatie role: platform-wide permission control
  • club_user pivot role: club-specific position display (admin/member label per club)
  • club_id scoping in all controller queries: prevents data leakage regardless of Spatie role

Full Fix (2–3 hours implementation)

Build a custom CheckClubRole middleware that checks the pivot table role for the specific club being accessed:

// middleware: clubrole:admin
$pivotRole = $user->clubs()
    ->where('club_id', $request->route('club'))
    ->first()?->pivot->role;

if ($pivotRole !== $requiredRole) {
    abort(403, 'You do not have the required role in this club.');
}

This would make permissions truly club-specific — Grace could be Admin in Club A and Member in Club B with different access rights in each.
